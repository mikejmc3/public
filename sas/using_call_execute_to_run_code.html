Using the CALL EXECUTE Routine to Run SAS Code

Experienced SAS programmers are well aware of the common practice of "using code to write code," most commonly using the macro interpreter. While I find SAS macros to be incredibly helpful tools for writing flexible code, there are some use cases where I rely on the CALL EXECUTE routine to add some needed flexibility to my programs.

In the example in this notebook, we will pretend that we want to perform complex transformations on inputs in a logistic regression model. We will use CALL EXECUTE to make the recoding process a little bit simpler.

A Primer on WOE Inputs

In this notebook, we will use a logistic regression model to attempt to classify vehicles in SASHELP.CARS as trucks/SUVs vs. other types. The data table contains a variety of inputs we could add to a prediction model, including both categorical and continuous values. For the purposes of this analysis, we will transform the inputs into weight of evidence (WOE) values.

A deep-dive into WOE is beyond the scope of this notebook, but for those who may be unfamiliar with this technique, understand that inputs are binned, and then those bins are transformed in such a way that allows the WOE-transformed values to be added to the model as continuous predictors. This technique is prevalent in risk modeling, as it allows for the creation of easy-to-interpret risk scorecards, which are highly valued for their transparency in relating the contribution of specific factors to an estimated risk score. For more information, see this page.

  • 0 = All other types

Bin Inputs

PROC CAS provides two different CAS actions to bin inputs, depending on whether they are categorical or continuous in nature.

Categorical Variables

Let's bin the categorical inputs first. Because I want larger, positive WOE values to be associated with trucks and SUVs, I will set the "event" to be 0 here. Remember that the focus of this notebook is on an interesting use of CALL EXECUTE, so I will not dwell much on what is going on here in PROC CAS. What is important for the reader to understand is that I am going to bin the DRIVETRAIN, MAKE, and ORIGIN categorical inputs, and I will calculate their WOE values. This CAS action will output several data tables, including one with bins and another with WOE values. We'll take a look at the outcomes for the DRIVETRAIN input.

* Perform binning and WOE transformation;
ods select vartransinfo;
proc cas;
    dataPreprocess.catTrans /
        table={
            caslib="casuser",
            name="cars"
        },
        casout={
            caslib="casuser",
            name="bins_cat",
            replace=true
        },
        copyvars={
            "row_id",
            "target",
            "drivetrain",
            "make",
            "origin"
        },
        inputs={
            "drivetrain",
            "make",
            "origin"
        },
        targets={"target"},
        events={"0"},
        method="woe",
        evaluationStats=true,
        casOutBinDetails={
            caslib="casuser",
            name="bin_details_cat",
            replace=true
        },
        casOutLevelBinMap={
            caslib="casuser",
            name="bin_map_cat",
            replace=true
        }
    ;
quit;

* Check DRIVETRAIN;
title "Bins and WOE values for levels of DRIVETRAIN";
proc sql;
    select
        a._variable_,
        a._binid_,
        a._level_,

Use CALL EXECUTE Routine to Create WOE Inputs

Our goal is to use the WOE values we calculated above as predictors in our model, but PROC CAS did not simply add the new WOE values to the model data. Instead, we got summary tables. There are several approaches we could take here, but I will demonstrate how we can save code to a text variable in a data table and then use the CALL EXECUTE routine to run that code within a DATA step to recode bin IDs to WOE values.

To make things easier to follow, I will break down the code generation into three distinct steps:

  1. Set up the DATA and SET statements.
  2. Generate the IF / THEN logic - where the magic happens.
  3. Set up the KEEP and RUN statements.

After that I will combine these three steps and run the code inside a _NULL_ DATA step using CALL EXECUTE.

Side note: We did the preceding steps in CAS, because that is a requirement for the CAS actions we used to do the binning and WOE transformation steps. We are now moving back to executing on the client in a V9 library (WORK), because that works best for CALL EXECUTE.

Categorical Inputs

* In all three steps, I will set up a variable I name CODE;
* Step 1 outputs the DATA and SET statements;
data work.recode_woe_cat_1;
    length code $ 500;
    code = "data work.woe_cat;";
    output;
    code = "    set casuser.bins_cat;";
    output;
run;

* Step 2 concatenates values from the source table to write IF/THEN statements;
data work.recode_woe_cat_2 (keep=code);
    set casuser.bin_details_cat;
    by _variable_ _binid_;
    length code $ 500;
    code = "    if cat_" || strip(_variable_) || " = " || strip(put(_binid_, 8.)) || " then woe_" || strip(lowcase(_variable_)) || " = " || strip(put(_woe_, 25.15)) || ";"; 
    * Do not pull in WOE output if only 1 bin (WOE will equal 0);
    if not (first._variable_ and last._variable_) then output;
run;

* Step 3 includes the KEEP and RUN statements;
data work.recode_woe_cat_3;
    length code $ 500;
    code = "    keep row_id woe_:;";
    output;
    code = "run;";
    output;
run;

* Run the combined steps using CALL EXECUTE;
data _null_;
    set
        work.recode_woe_cat_1
        work.recode_woe_cat_2
        work.recode_woe_cat_3
    ;
    call execute(code);
run;
sas

Put It All Together

Let's join our binned and WOE-transformed inputs onto our base data table that we will use in modeling. I am going to use hash objects to merge four tables onto my base table in a single DATA step.

data work.cars_woe;
    if 0 then set
        work.cars
        casuser.bins_cat (keep=cat_:)
        casuser.bins_cont (keep=dis_:)
        work.woe_cat (keep=woe_:)
        work.woe_cont (keep=woe_:)
    ;

    if _n_ = 1 then do;
        declare hash h_b_cat(dataset: "casuser.bins_cat", duplicate: "error");
        h_b_cat.definekey("row_id");
        h_b_cat.definedata("cat_drivetrain", "cat_make", "cat_origin");
        h_b_cat.definedone();

        declare hash h_b_cont(dataset: "casuser.bins_cont", duplicate: "error");
        h_b_cont.definekey("row_id");
        h_b_cont.definedata("dis_enginesize", "dis_horsepower", "dis_mpg_city", "dis_msrp", "dis_weight");
        h_b_cont.definedone();

        declare hash h_w_cat(dataset: "work.woe_cat", duplicate: "error");
        h_w_cat.definekey("row_id");
        h_w_cat.definedata("woe_drivetrain", "woe_make", "woe_origin");
        h_w_cat.definedone();

        declare hash h_w_cont(dataset: "work.woe_cont", duplicate: "error");
        h_w_cont.definekey("row_id");
        h_w_cont.definedata("woe_enginesize", "woe_horsepower", "woe_mpg_city", "woe_msrp", "woe_weight");
        h_w_cont.definedone();
    end;

    set work.cars;

    call missing(of cat_: dis_: woe_:);

    rc1 = h_b_cat.find();
    rc2 = h_b_cont.find();
    rc3 = h_w_cat.find();
    rc4 = h_w_cont.find();

    drop rc:;
run;
sas

Fit a Model

OK, this was our ultimate end goal in this exercise. Again, the purpose of this notebook is to illustrate the use of CALL EXECUTE, but why stop now? I am going to cut a few corners here and not bother with partitioning, cross-validating, building an actual scorecard from the output, or anything else too fancy. The main goal here is to demonstrate that the model takes in the WOE-transformed inputs as continuous predictors, as we intended. This allows the resulting model to have useful properties when it comes time to interpret the results, but we'll get into that in more detail on another day.

ods select ModelInfo NObs ResponseProfile ModelBuildingSummary ParameterEstimates OddsRatios Association ROCCurve ROCOverlay;
proc logistic data=work.cars_woe plots(only)=roc;
    model target (event="1") = woe_: / selection=stepwise;
run;
quit;
sas

Final Thoughts

In this notebook, we see how values stored in data tables can be concatenated with text to produce data-driven code that is run using the CALL EXECUTE routine. This routine, in addition to the SAS macro utility, provides SAS programmers with a useful tool to create and execute flexible code.
