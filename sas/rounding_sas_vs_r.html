Rounding Comparison: SAS vs. R

When I worked as a psychometrician, one of my key responsibilities was scaling and equating scores for large-scale standardized tests. The company I worked for at the time had a policy that required 100% independent replication of end-to-end results by a peer for all scaling and equating work. Under no circumstances could a pair of employees share code or specific details about their work, but one topic we were permitted to discuss was to agree upon a reasonable level of numerical precision for comparisons of results at various milestones. So my project buddy and I might agree to ensure our results match to the fifth decimal place, for example.

These were long, complicated workflows that involved hundreds of individual steps. There were numerous opportunities for two independent processes to get off track from one-another and arrive at different final answers. Fortunately, we all used the same software, at least: SAS. Over time, we began to explore alternatives to SAS, including R, which opened up some interesting new challenges to producing matching results, one of which turned out to be rounding of all things.

Rounding is a topic I must admit I have given little thought since I first learned about it in elementary school. If rounding to the nearest integer, a value of 4.4 rounds down to 4, a value of 4.6 rounds up to 5, and a "tie" value like 4.5 rounds up to 5. Right? Well, it turns out that depends.

Rounding in SAS

Let's generate a sequence of values in SAS ranging from 0.5 to 9.5, incrementing by 1. Let's use the ROUND function to round each value to the nearest integer. As demonstrated below, SAS always rounds a value of X.5 up to the next higher integer, so 1.5 rounds up to 2, 2.5 rounds up to 3, and so on.

data work.round;
    format
        x_raw 8.1
        x_round 8.
    ;
    do x_raw = 0.5 to 9.5 by 1;
        x_round = round(x_raw);
        output;
    end;
run;

* Display values;
title "Rounding in SAS";
proc print data=work.round;
run;
title;
sas

Rounding in R

I generated the same sequence of values in R and rounded them. As shown in the output, R rounds values of X.5 to the nearest even integer, so 1.5 rounds up to 2, 2.5 rounds down to 2, 3.5 rounds up to 4, and so on.

Code

x_raw <- seq(0.5, 9.5, 1)
x_round <- round(x_raw)

# Combine into a data frame and display
x_df <- data.frame(cbind(x_raw, x_round))
print(x_df)

Impact

Because rounding is a concept I simply took for granted, this discovery that SAS and R take different approaches to rounding came as a surprise. Independently replicating complex workflows is already challenging, and curveballs like this don't make the job any easier.

SAS uses the same strategy that I was taught in school: ties always round up, but R takes a more nuanced approach that rounds ties up or down, wherever the nearest even integer may be. Why does R do it this way? Let's compare means of unrounded versus rounded values to learn more.

Our expectation is that the mean of the rounded values should be similar to the mean of the original, unrounded values.

Means: SAS

proc means data=work.round mean maxdec=3;
    var x_raw x_round;
run;
sas

In SAS, the mean of the rounded values is 5.5, and the mean of the original, unrounded values in 5.0. Rounding has resulted in a positively biased mean for these values.

In R, the mean of the rounded values matches the mean of the original, unrounded values.

Why Is R Different?

Why does R round ties to the nearest even integer instead of always rounding up? As demonstrated in the above examples, the answer is bias. Always rounding up introduces positive bias into the rounded numbers. As noted on this page, R's rounding rules conform to the IEEE 754 technical standard (as does Python, by the way). Reducing bias is obviously a good thing, but there are some nuances to this practice that may create some unexpected results.

Gotchas

It is important to recognize that some decimal numbers cannot be precisely stored in a binary format, and therefore rounding to the closest even digit may sometimes lead to unexpected results. Let's see what happens when we round a sequence of numbers to the first digit.

Provide a Vector of Values

First, let's manually enter a vector of values (1.05, 1.15, 1.25) and round them to the first digit.

Code

x_raw_2 <- c(1.05, 1.15, 1.25)
x_round_2 <- round(x_raw_2, 1)
x_df_2 <- data.frame(cbind(x_raw_2, x_round_2))
print(x_df_2)

Output

> print(x_df_2)
  x_raw_2 x_round_2
1    1.05       1.0
2    1.15       1.1
3    1.25       1.2

Notice how the value of 1.15 rounds down to 1.1 instead of rounding up to the expected value of 1.2. This seems especially odd because it gives the impression that it breaks both the "nearest even" rounding rule that R uses, but also the simpler "always round up rule." The next higher value, 1.25, rounds down to 1.2 as we would expect.

Use a Sequence

Now, let's use the seq function to generate the same values (1.05, 1.15, 1.25) and once again round them to the first digit.

Code

x_raw_3 <- seq(1.05, 1.25, 0.1)
x_round_3 <- round(x_raw_3, 1)
x_df_3 <- data.frame(cbind(x_raw_3, x_round_3))
print(x_df_3)

Output

> print(x_df_3)
  x_raw_3 x_round_3
1    1.05       1.0
2    1.15       1.2
3    1.25       1.2

This time, all three values round as we would expect, including 1.15 now rounding up to 1.2.

It is important to keep in mind that how a number is stored and used in computations may not always directly reflect what is displayed to the user, which can sometimes lead to counter-intuitive outcomes like this. To further complicate matters, rounding outcomes may vary depending on the mechanism used to produce the original number that gets rounded.

Let's compare individual values to see if R treats them as equal.

Code

x_vec <- c(1.05, 1.15, 1.25)
x_seq <- seq(1.05, 1.25, 0.1)

# Display values
print(x_vec)
print(x_seq)

# Compare values
print(x_vec == x_seq)

Replicate in SAS

Let's see how SAS rounds these values. For consistency, we will manually enter one set of values, and we'll use a loop to generate the other set.

* Manual entry;
data work.round_2;
    input x_raw 8.2;
    format x_round 8.1;
    x_round = round(x_raw, 0.1);
    datalines;
1.05
1.15
1.25
;
run;

* Generate in a loop;
data work.round_3;
    format
        x_raw 8.2
        x_round 8.1
    ;
    do x_raw = 1.05 to 1.25 by 0.1;
        x_round = round(x_raw, 0.1);
        output;
    end;
run;

* Display values;
title "Rounding in SAS: Manual Entry";
proc print data=work.round_2;
run;

title "Rounding in SAS: Loop";
proc print data=work.round_3;
run;
title;
sas

A More Realistic Impact Assessment

Encounters with exactly perfect ties in need of rounding are exceptionally rare in my experience. In the vast majority of cases, I am dealing with numbers with many decimal places, or integers, or some combination.

So how much of a concern is this rounding rule in a more realistic setting? Let's take a look.

* Generate 100 replications with 10,000 rows each;
* Simulate values from a normal distribution with a mean of 100 and SD of 15;
* Round the simulated values, and compare the means from each replication;
data work.sim;
    call streaminit(42);
    do rep = 1 to 100;
        do obs = 1 to 10000;
            x_raw = rand("normal", 100, 15);
            x_rounded = round(x_raw);
            output;
        end;
    end;
run;

proc summary data=work.sim nway;
    class rep;
    var x_raw x_rounded;
    output out=work.sim_means mean=mean_raw mean_rounded;
run;

data work.plot_sim;
    set work.sim_means;
    mean_diff = mean_rounded - mean_raw;
run;

title "Mean Differences (Unrounded vs. Rounded) Across 100 Replications";
proc sgplot data=work.plot_sim;
    needle x=rep y=mean_diff;
    yaxis grid;
run;
title;
sas

The plot above compares the means of the rounded values (using the SAS "always round ties up" rule) to the means of the original, unrounded values across 100 replications. There is no clear evidence of bias associated with rounding in this more realistic scenario. This makes sense because only perfect ties would lead to bias, and we expect those values to be rare.

Conclusions

I have demonstrated that rounding works fundamentally differently in SAS and R. SAS uses a more traditional rule that always rounds ties up, while R uses a rule that usually rounds ties to the nearest even value, but limitations with the numerical representation of certain floating point numbers may sometimes lead to situations where rounding in R appears to contradict expectation. In my experience, exact ties are rare enough that I am not concerned about the potential for rounding to introduce bias into a distribution's mean. The greatest concern here would be attempting to compare a set of numbers rounded in SAS to the same set of numbers rounded in R.
