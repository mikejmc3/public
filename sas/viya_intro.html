A Basic Introduction to Cloud Analytic Services on SAS Viya

Approaching SAS Viya as an experienced SAS 9 programmer presented me with a few changes that caused some initial confusion. Not only did I have to learn a few new habits, but I had to unlearn a couple as well. Fortunately, after a brief adjustment period I was able to get up to speed and start taking advantage of what SAS Viya has to offer.

This notebook contains some examples of key concepts that may benefit SAS 9 programmers with little or no prior experience with SAS Viya. This is by no means a comprehensive guide, but rather a gentle introduction with some attention paid to areas I found to be particularly mystifying when I was first familiarizing myself with the platform.

Connecting to CAS

The CAS statement initiates a Cloud Analytic Services (CAS) session. I sometimes think of SAS Viya as having two "brains." There is the legacy "V9 brain," which is what you use when you work with legacy *.sas7bdat data sets in V9-engine SAS libraries (e.g., WORK), and the "CAS brain" is what you use to work with modern in-memory data tables loaded into CASLIBs.

In SAS 9, you start a new SAS session immediately when you launch SAS (in the case of SAS Display Manager) or when you first submit code and connect to the SAS compute server (in the case of SAS Enterprise Guide).

In SAS Viya, your SAS session begins when you launch SAS Studio, but you have to submit CAS statement code to connect to a CAS server and start your CAS session. Depending on how your SAS Viya environment is configured, you may have several available CAS servers for use. It is possible to have multiple CAS sessions running concurrently and in use by the same program. In this notebook, we will use a single session.

Minimally, the CAS statement contains the following:

  1. CAS statement keyword,
  2. A unique name for your CAS session,
  3. Correct host name (obtain from your SAS Viya system administrator).

cas mycas host="my-viya-server-name";

Other options are available, as shown below. For security reasons, I will not include my SAS Viya environment's host name in the committed code for this notebook.

* Start a CAS session named CASMAIN;
cas casmain
    host="/* Provide your host name here */"
    port=5570
    sessopts=(
        timeout=1800
        locale="en_US"
        metrics=false  /* Set to TRUE for more details in the log */
    )
;
sas

Libraries

SAS Viya users are able to work with CAS data sources using traditional SAS programming techniques (DATA and most PROC steps). In order to use these to work with CAS data, you must use a LIBNAME statement to associate a LIBREF with the CASLIB that contains your CAS data.

Experienced SAS programmers will be familiar with the concept of LIBNAME engines. Just like we use the V9 engine to work with SAS 9 data sets and the XLSX engine to work with Microsoft Excel files, we use the CAS engine to work with CAS data. The LIBNAME statement for CAS data takes the following form.

libname mylib cas caslib="caslib-name" sessref=cas-session-name;

Breaking the above statement down, we see that it contains the following:

  1. LIBNAME statement keyword,
  2. A unique LIBNAME (standard LIBNAME rules apply!) for this SAS library,
  3. CAS keyword indicating this LIBNAME uses the CAS engine,
  4. The name of the associated CASLIB (enclose in quotes as shown above if the CASLIB name contains spaces), and
  5. The name of the CAS session that corresponds to the target CASLIB.

Note that the SESSREF= argument is not strictly required, but I strongly encourage users to always specify it. The user has the option to have several active CAS sessions running concurrently, so being explicit about which LIBNAME corresponds to which CAS session can save the programmer some headaches. If the scope of the program expands over time to include multiple CAS sessions, including this option prevents previously-working code from blowing up when LIBNAME references become ambiguous, so I find it is best to always define the CAS session.

Let's set up a LIBNAME connected to the CASUSER CASLIB. For simplicity, we will call our new LIBNAME CASUSER, but we could just as easily name this something else if we wanted to.

libname casuser cas caslib=casuser sessref=casmain;
sas

How Do I Know Which CASLIBs Are Available?

You may be thinking, but how did you know there was a CASLIB called CASUSER to connect to?

To answer this question, one option is to go to SAS Data Explorer in your SAS Viya environment, expand the server that you wish to use, and you will see a list of CASLIBs.

If you prefer to go the code route, you can submit code like this to see a list of CASLIBs in the log.

caslib _all_ list sessref=cas-session-name;

Note: You may not have read access to every CASLIB that you see, depending on your environment's security configuration.

Using the DATA Step to Read and Write Tables

Let's grab some data stored in a V9 SAS library and copy it over to CAS. We will use the trusty SASHELP.CARS table in these examples. Note that in SAS Viya, data tables in SASHELP use the legacy V9 engine.

* Copy a table from V9 to CAS;
data casuser.cars;
    set sashelp.cars;
run;
sas

The above step read the CARS table from SASHELP and wrote it to CASUSER on CAS. Here, we can do the typical operations on our data table using traditional SAS code, even though our data table is in CAS.

* Create a new CAS table from another CAS table;
data casuser.cars_altered;
    set casuser.cars;
    mpg_city_sq = mpg_city ** 2;
    length type_up $ 8;
    type_up = upcase(type);
run;
sas

Using the DATA step, we can easily move our data from CAS to the legacy V9 engine WORK library, too.

data work.cars;
    set casuser.cars;
run;
sas

Using PROC CASUTIL to Read and Write CAS Tables

The CASUTIL procedure is a very helpful tool for managing CAS data. Somewhat similar to PROC DATASETS, it is a bit of "Swiss Army Knife" among CAS tools, and I find it useful for a variety of use cases--mostly for deleting, copying, loading, saving, and/or promoting data (see further below for more on these concepts).

Let's use PROC CASUTIL to load SASHELP.CARS into the CASUSER CASLIB. Name the new table CARS_CU.

* Copy SASHELP.CARS to the CASUSER CASLIB, and name that new table CARS_CU;
proc casutil sessref=casmain;
    load  /* Load table into memory - see further below for more details */
        data=sashelp.cars  /* Input V9 LIBNAME and table name */
        outcaslib="casuser"  /* Output CASLIB. Note: use the CASLIB and *NOT* the LIBNAME. In this case, they share the same name */
        casout="cars_cu"  /* Output CAS table name */
    ;
quit;
sas

We can use PROC CASUTIL to read and write in-memory tables as well. Let's create a new copy of the CARS_CU table and name the new copy CARS_CU2. Both will live in the CASUSER CASLIB.

proc casutil sessref=casmain;
    copy  /* Use the COPY statement to copy in-memory tables */
        incaslib="casuser"
        casdata="cars_cu"
        outcaslib="casuser"
        casout="cars_cu2"
    ;
quit;
sas

Note that, by default, PROC CASUTIL will not let you overwrite an existing CAS table. Let's re-run this code and try to overwrite CARS_CU2.

* This will cause an error because CARS_CU2 already exists;
proc casutil sessref=casmain;
    copy
        incaslib="casuser"
        casdata="cars_cu"
        outcaslib="casuser"
        casout="cars_cu2"
    ;
quit;
sas

This action is refused, and the user sees the following message in the log.



ERROR: The table cars_cu2 already exists in the session.
ERROR: The action stopped due to errors.
NOTE: The Cloud Analytic Services server processed the request in 0.014585 seconds.
ERROR: The copying of table "cars_cu2" in the "casuser" caslib failed.

We can get around this by adding the REPLACE option. This works only for local or session scope tables. See further below for more information on table scope.

* This will work now;
proc casutil sessref=casmain;
    copy
        incaslib="casuser"
        casdata="cars_cu"
        outcaslib="casuser"
        casout="cars_cu2"
        replace  /* Replace table if it already exists */
    ;
quit;
sas

Note that by contrast, the DATA step does not require any special option to overwrite an existing in-memory table.

* Use DATA step to overwrite CAS table;
data casuser.cars_cu2;
    set casuser.cars_cu;
run;
sas

Using PROC CAS to Read and Write CAS Tables

To me, PROC CASUTIL feels rather similar to other "traditional" SAS code, with the main difference being splitting out CASLIBs and table names into separate arguments as opposed to using the familiar LIBNAME.TABLE format generally used elsewhere in SAS programming.

PROC CAS, on the other hand, feels more like a new language unto its own. I do not make heavy use of PROC CAS in my day-to-day work. It provides a level of depth that other, more high-level procedures cannot always match, though. Most of the time, I rely on simpler options like the DATA step or PROC CASUTIL, and I resort to PROC CAS for more advanced use cases.

Below, you will see an example of using PROC CAS to make a copy of a CAS table.

A note on PROC CAS syntax: under this procedure, specific tasks are performed by actions, which are grouped within action sets. These are typically called in the following way:

proc cas;
    actionSet.action / parameter={argument1="value", argument2="value", ...};
quit;

Parameter arguments are grouped within curly braces and specific arguments for the parameter appear somewhat similar to comma-separated lists matching arguments to values. Most provided values are enclosed in quotes, with some exceptions.

In the following example, we will use the copyTable action within the table action set to make a copy of the CARS_CU table we previously created using PROC CASUTIL, and we will name the new table CARS_PC. We will include a REPLACE option to allow an overwrite if the table already exists.

proc cas;
    session casmain;  /* Notice that we use SESSION here instead of SESSREF, no equals, no quotes */

    table.copyTable /
        table={  /* Input arguments */
            caslib="casuser",  /* Input CASLIB */
            name="cars_cu"  /* Input table name */
        },

        casout={  /* Output arguments */
            caslib="casuser",  /* Output CASLIB */
            name="cars_pc",  /* Output table name */
            replace=TRUE  /* Notice no quotes around Boolean TRUE this time */
        }
    ;
quit;
sas

The Key Benefit of CAS: Speed

CAS data tables are stored in memory and distributed among several worker nodes. Specific details, including the number of available worker nodes, are unique to a given environment's configuration.

Distributed, in-memory data are ideal for parallel operations, which can greatly improve execution time when dealing with large data tables.

Conversely, legacy V9 *.sas7bdat data tables are saved to disk and accessed sequentially (one row at a time), which generally translates to slower execution time when working with large data tables.

Let's use a loop to expand the CARS table (N = 428) to have just over 10 million rows and do an informal performance comparison between CAS and V9.

CAS Data

* Stack data;
data casuser.big_cars;
    set casuser.cars;
    do i = 1 to 23365;
        output;
    end;
    drop i;
run;
sas
* Use CAS-enabled FREQTAB procedure;
proc freqtab data=casuser.big_cars;
    tables type;
run;
sas

Reflections on CAS vs. V9

OK, CAS did execute faster, but V9 performance is still very fast, and we're measuring in fractions of seconds you may be thinking right now. That is true, but these are very simple examples, and performance differences between CAS and V9 can get quite noticeable when performing more complex operations.

Furthermore, the performance gulf between CAS and V9 widens further once you take into account that sorting is no longer needed in CAS. Traditional V9 SAS data tables are stored on a disk, and the rows have an inherent ordering to them. In a distributed, in-memory environment like CAS, a concept like row ordering no longer makes sense. How do you decide which row is "first" when the data are distributed across multiple nodes?

Because row ordering as a concept stops making sense in CAS, sorting goes away with it. Experienced SAS 9 programmers are familiar with the standard practice of sorting data before merging in a DATA step. SAS Viya programmers get the luxury of skipping the sorting and proceeding right to the merging. Eliminating computationally expensive sorting steps from workflows can yield substantial performance gains.

Necessary Adjustments

Unfortunately, the move from V9 to CAS is not all sunshine and rainbows. Although we happily bid goodbye to some relics like sorting, some tradeoffs are a part of the bargain. Not everything I list below is necessarily a downgrade, but these differences may trip up SAS Viya novices.

CAS and SQL

Although my personal bias is to rely on traditional SAS DATA and PROC steps for most data manipulation, I know some SAS users really, really love PROC SQL, and I have seen some very impressive (if somewhat head-spinning) queries that do a whole lot in a single step. Unfortunately, at the time that I am writing this, SAS Viya does not allow PROC SQL to read from CAS and write back to CAS in a single step.

It is possible to use PROC SQL when both input and output destinations are V9, or if one destination is CAS and the other is V9, but for reasons I do not fully understand, using PROC SQL is forbidden when both input and output destinations are CAS.

* This will not work;
proc sql;
    create table casuser.cars_sql as
    select * 
    from casuser.cars;
quit;
sas

The log provides the following feedback:



ERROR: PROC SQL does not support CREATE TABLE AS statements where both the source and destination include SASVIYA libname references.

SAS does offer PROC FEDSQL as an option for users who wish to read from and write to CAS in a single SQL query, but I find this procedure's strict adherence to the ANSI SQL:1999 core standard to be heavy-handed and its log messages generally to be vague and unhelpful when errors occur (and believe me, you will see them often). To add to my list of complaints, odd limitations are imposed on PROC FEDSQL on SAS Viya. One example is that subqueries can be used with certain other data sources, but not with CAS data. I tend to avoid PROC FEDSQL for those reasons.

Data Complications

In SAS 9, data management is pretty straightforward. You save data into a file with a *.sas7bdat file extension. That file lives somewhere on your computer or on a server, depending on your configuration. You specify a LIBNAME as a shortcut of sorts to the folder where your data file is saved. You reference tables in a LIBNAME.TABLE format. Pretty easy.

In SAS Viya, things get a bit more complicated. We already talked a bit about sessions, servers, and CASLIBs, but the differences go even deeper than that. In general, I would say that data gets more complicated in SAS Viya, which increases the learning curve. It's not awful, but there are a few more concepts to mentally juggle when dealing with data in CAS.

Revisiting Established Concepts

First, let's revisit some topics I already mentioned above. Depending on your specific system configuration, when you log onto SAS Viya, you may have multiple available servers. If desired, you can set up separate CAS sessions to connect to each and every one of these servers, if you like. You could even have multiple sessions connected to the same server if you like, although there would be little practical value in doing that.

CAS data lives in what are called CASLIBs on each server. Most (but not all, see below) SAS code needs you to assign a LIBNAME to the desired CASLIB before you can work with the table in SAS code.

New Data Concepts

SAS Viya was built to permit data to be accessed flexibly by users with varying access permissions in a cloud environment. To accommodate this goal, some new concepts had to get added to the old SAS 9 model.

Scope and Promotion

CAS tables now include the concept of scope, which may be either local or global. When I first create a CAS data table, it has local (or sometimes referred to as session) scope, which means only I can see and use it. Even if I save my data table to a CASLIB that others can access, nobody but me will be able to see or access my data table until I make it visible to them. This can get a little confusing when moving between SAS Viya applications. I may be able to see and use a local-scope table in SAS Studio, but when I switch over to SAS Visual Analytics (still logged in as myself), I won't be able to see the table there. This is because local scope doesn't just limit access to the user, but to a session, and Visual Analytics is using a different session than SAS Studio.

The process that allows local-scope data tables to be visible and accessible to other users and sessions is called promotion. Once I promote a data table, it goes from local scope to global scope. A table with global scope can be seen and accessed by any other user or session (including me in SAS Visual Analytics), provided they meet all access requirements.

Saving and Loading - Not What They Seem

Words like "load" and "save" have very specific connotations in computing, and this goes back decades. Loading involves reading and saving involves writing. SAS Viya uses these terms differently than most users would likely expect. In SAS Viya, both involve outputing or creating something, but the difference depends on the type of thing you create.

First of all, CAS handles data very differently than SAS 9. Whereas SAS 9 reads/writes *.sas7bdat files that are saved to a disk, SAS Viya applications like SAS Studio and SAS Visual Analytics use in-memory data.

In SAS Viya, the term "load" refers to loading data into memory.

So, when you read a V9 data table from disk (e.g., reading a table in WORK) and you copy it to a CASLIB in a DATA step, you are actually loading that table into memory when you do that. Everything important involving CAS data in SAS Viya uses in-memory data. Whether you are running code, building an interactive dashboard, or training a model, you are using in-memory data in each case.

Everything stored in memory is temporary. Switch off the power, and it's gone with a poof! For this reason, it is important to have backup files stored, and this is where the "save" part comes in.

In SAS Viya, the term "save" refers to saving a backup copy of data.

Should something unforeseen happen that corrupts or deletes the in-memory data, its backup file (usually saved in *.sashdat or *.csv format) is used to reload the data back into memory. The backup file serves only to reload data back into memory. No code, reports, models, etc. make direct use of the backup file, with the lone exception being code used to create, overwrite, or delete a backup file.

Reloading Data from a Backup File

Let's say we have an in-memory table and a backup file saved in a CASLIB. For whatever reason, the in-memory table gets unloaded from memory. What happens? Confusingly, different SAS Viya applications handle this situation differently.

If the table in question is connect to a SAS Visual Analytics report, as soon as a user opens that report, SAS Viya will detect that the table is not currently loaded into memory, and it will reload the table from its backup file automatically to display the report.

SAS Studio is not so kind. If a user runs code that references a CAS table that is not currently loaded into memory, the code will crash and SAS Studio will raise an error. It is up to the user to reload the table before running code.

SAS Viya users must keep differences like this in mind if they regularly use both applications.

Row Level Security

I will not go into great detail here, but SAS Viya allows tremendous flexibility when granting users access to data. Data access in SAS Viya is not "all or nothing" but rather, users can be restricted to see and access only portions of data based on their row level security settings. For example, if a shared data table contains an organization's worldwide sales numbers, the table's creator can place different restrictions on different users. Frontline Sales staff can be limited to seeing only their assigned accounts, regional Sales managers can see all accounts in their region, and Executive leadership can see the global view.

The tradeoff for this added flexibility is added complexity when granting and managing user access.

Exceptions to Rules

I previously said that SAS programmers need to use a LIBNAME when using "traditional" SAS DATA and PROC steps to access CAS data. The word "traditional" here implies most SAS procedures, but there are certain exceptions. Most notably, PROC CASUTIL, PROC CAS, and PROC FEDSQL will use the CASLIB instead of the LIBNAME when dealing with CAS data. Users need to be mindful of which procedure they are using and reference the appropriate LIBNAME or CASLIB accordingly.

Saving and Promoting

So far, all tables I loaded into CAS were local scope tables, meaning they were accessible only to me in the present session. If I wanted to share my data table with anyone else, including myself in another application like SAS Visual Analytics, I need to promote the data to global scope. Furthermore, because data loaded into memory could become corrupted or get deleted with no recourse, we will save a backup file that can be used to reload the in-memory data, if needed.

I generally rely on PROC CASUTIL to load, promote, and/or save CAS data tables.

Note: The CASUSER location we have been using for convenience in these examples is not ideal for sharing in practice. This CASLIB is specific to me as an individual user. In the real world, using a team/shared CASLIB would be preferable, but we are just demonstrating code here.

Promote and Save an Existing CAS Table

Let's start by promoting a local scope CAS table to global scope. We will save a backup *.sashdat file for this table as well.

Important: Once a table is promoted to global status, it is protected from being overwritten easily. A promoted table must be purposefully unloaded instead of being overwritten.

proc casutil sessref=casmain;
    /* Promote CAS data and rename promoted version */
    promote
        /* Input CASLIB and table */
        incaslib="casuser"
        casdata="cars_cu"
        keep  /* Keep the local scope version of the table after promoting */
        /* Rename promoted copy for fun */
        outcaslib="casuser"
        casout="cars_promoted"
    ;

    /* Save a sashdat backup file from the promoted in-memory table */
    save
        /* Input CASLIB and in-memory data */
        incaslib="casuser"
        casdata="cars_promoted"
        /* Output CASLIB and file name */
        /* SAS will automatically add .sashdat file extension if none provided */
        /* Linux file names are CASE SENSITIVE */
        outcaslib="casuser"
        casout="cars_promoted"
        replace  /* Replace cars_promoted.sashdat if it already exists */
    ;
quit;
sas

Load, Promote, and Save V9 SAS Data Table

Here is how we load a V9 SAS data table into CAS, promote it, and save a backup file.

proc casutil sessref=casmain;
    /* Load V9 data to CAS */
    load 
        /* Input V9 LIBNAME.table - no quotes here */
        data=sashelp.cars
        /* Output CASLIB and table name */
        outcaslib="casuser"
        casout="cars_v9_promoted"
        promote  /* Promote the table once loaded */
    ;

    /* Save a sashdat backup file from the promoted in-memory table */
    /* This looks similar to the previous example */
    save
        /* Input in-memory table created above in LOAD statement */
        incaslib="casuser"
        casdata="cars_v9_promoted"
        /* Output backup file details */
        outcaslib="casuser"
        casout="cars_v9_promoted"
        replace
    ;
quit;
sas

Unloading In-Memory Data and Deleting Backup Files

Let's say we are done using our data, so we wish to remove it. Recall that there is both an in-memory data table and a *.sashdat backup file. Here is how we remove those.

* Table #1;
proc casutil sessref=casmain;
    /* Unload data from memory */
    droptable
        incaslib="casuser"
        casdata="cars_promoted"
        quiet  /* Do not throw an error if the table is not currently loaded into memory */
    ;

    /* Delete backup file */
    deletesource
        incaslib="casuser"
        casdata="cars_promoted.sashdat"  /* CASE SENSITIVE, and include file extension here */
        quiet  /* Again, do not throw an error if file not found */
    ;
quit;

* Table #2;
proc casutil sessref=casmain;
    /* Unload data from memory */
    droptable
        incaslib="casuser"
        casdata="cars_v9_promoted"
        quiet
    ;

    /* Delete backup file */
    deletesource
        incaslib="casuser"
        casdata="cars_v9_promoted.sashdat"
        quiet
    ;
quit;
sas

Terminating CAS Sessions

Although your CAS session will eventually time out and terminate on its own, closing your CAS session when you are done is considered best practice. The following code ends all active CAS sessions.

cas _all_ terminate;
sas

Over time, I overcame these challenges and now I do most of my day-to-day work in SAS Viya without any issue.
