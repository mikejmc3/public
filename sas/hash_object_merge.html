Hash Object Merges in SAS

SAS provides users with several options to join data tables. Arguably the two most commonly-used methods among SAS programmers include:

  • MERGE statement within the DATA step, or
  • PROC SQL join.

Let's examine some pros and cons of these traditional approaches, and let's consider an alternative method: using hash objects to merge data.

Merging Data Tables Using the MERGE Statement

Clean Merge

In SAS 9, both data tables must be sorted by the variable(s) to be used in the merge's BY statement prior to performing the merge. This requirement does not apply to Cloud Analytic Services (CAS) data tables in SAS Viya.

* Sort data tables ------------------------------------------------------------;
proc sort data=work.clean_1 out=work.clean_1_s;
    by id x;
run;

proc sort data=work.clean_2 out=work.clean_2_s;
    by id y;
run;

* Demonstrate different types of joins ----------------------------------------;
* Inner join: keep only rows in both tables;
data work.merge_ds_clean_1;
    merge
        work.clean_1_s (in=in1)
        work.clean_2_s (in=in2)
    ;
    by id;
    * Output only rows that appear in both tables;
    if in1 and in2 then output;
run;

title "DATA Step Merge: Inner Join";
proc print data=work.merge_ds_clean_1;
run;
title;

* Left join: keep all rows in first (or left) table;
data work.merge_ds_clean_2;
    merge
        work.clean_1_s (in=in1)
        work.clean_2_s (in=in2)
    ;
    by id;
    * Output only rows that appear in first table;
    if in1 then output;
run;

title "DATA Step Merge: Left Join";
proc print data=work.merge_ds_clean_2;
run;
title;

* Outer join: keep all rows that appear in either table;
* Note that this is the default MERGE statement behavior;
data work.merge_ds_clean_3;

    ;

Joining Tables Using PROC SQL

Clean Join

A PROC SQL join has one significant advantage over using the MERGE statement in a DATA step: there is no requirement in SAS 9 to sort the tables prior to performing the join. This does not mean that no sorting occurs, however. PROC SQL simply sorts the tables automatically when performing the join rather than relying on the user to perform the sort first. So we might say that PROC SQL joins are a little more efficient in terms of code because a prior PROC SORT step is not required. Both a DATA step merge and a PROC SQL join involving sorting, however, which can be computationally costly and therefore somewhat slow for large data tables. Efficiency in terms of execution time is likely similar across these two traditional methods.

The above considerations apply to SAS 9. Considerations get a bit more nuanced in SAS Viya. As previously noted, DATA step merges using CAS data in SAS Viya do not require sorting. PROC SQL with CAS data in SAS Viya has some restrictions, though. Notably, PROC SQL does not permit reading from CAS and writing back to CAS in the same step.

* For simplicity, we will just display output instead of creating tables;

* Inner join: keep only rows in both tables;
title "PROC SQL Inner Join";
proc sql;
    select 
        a.id, 
        a.x, 
        b.y
    from
        work.clean_1 a
        inner join
        work.clean_2 b
        on a.id = b.id
    ;
quit;
title;

* Left join: keep all rows in first (or left) table;
title "PROC SQL Left Join";
proc sql;
    select 
        a.id, 
        a.x, 
        b.y
    from
        work.clean_1 a
        left join
        work.clean_2 b
        on a.id = b.id
    ;
quit;
title;

* Outer join: keep all rows in both tables;
title "PROC SQL Outer Join";
proc sql;
    select 
        coalesce(a.id, b.id) as id, 
        a.x, 
        b.y
    from
        work.clean_1 a
        full outer join
        work.clean_2 b
        on a.id = b.id
    ;
quit;
title;

Now let's see what happens when there are duplicate ID values in both tables when performing a full outer join using PROC SQL.

        a.x, 
        b.y
    from
        work.dup_1 a
        full outer join
        work.dup_2 b

    order by id, x, y

Notice the difference in behavior here compared to what we saw in the DATA step. When using PROC SQL to join two tables by a variable that has duplicate rows in both, we get something resembling a Cartesian join for the duplicate ID. In this case, every value of X is matched against every value of Y for all rows where ID = 2. I would argue that this behavior is more desirable than the DATA step's somewhat chaotic approach, but I would still prefer a warning or error message appearing in the log when a many-to-many join occurs, if only because two rows of ID = 2 in one source table and three rows of ID = 2 in the other source table have now expanded to six rows in my output table, which I may not have expected. No special log message, not even a note, is provided to the user when performing a many-to-many join using PROC SQL.

Hash Object Merges

I find hash object merging useful because it solves two key issues that I identified in the preceding examples.

First, no sorting is required (once again, this impacts SAS 9 and not CAS data in SAS Viya). Unlike traditional DATA step merges, there is no need to perform potentially time-consuming PROC SORT steps prior to merging data tables. And unlike PROC SQL, hash object merging does not perform a "sneaky sort" either, whereby it sorts data (and thus potentially slows down your workflow) without telling the user as part of performing the merge. No prior sorting is required, and no unannounced quiet sorting is performed as part of the merge, either.

Second, the user has the option to get an error message if duplicate merge key values are present. As noted in previous examples, duplicate values in a merge key can lead to unpredictable outcomes, which can create headaches if this situation goes unnoticed. I find error messages very hard to overlook, so I appreciate this option.

Clean Merge

Let's look at some examples of clean merges using this method. Note that the input tables we use in these examples are not sorted.

* Inner join: keep only rows in both tables -----------------------------------;
data work.merge_hash_clean_1;
    * Get variable characteristics;
    if 0 then set work.clean_1 work.clean_2;

    * Load WORK.CLEAN_2 data into a hash object named H;
    * The merge key is ID;
    * Pull in variable Y from the hash object into the merged output table;
    * Note that table/variable name arguments are enclosed in quotes;
    * If there are multiple arguments, they are separated by commas;
    if _n_ = 1 then do;
        declare hash h(dataset: "work.clean_2", duplicate: "error");
        h.definekey("id");  * Merge key variable;
        h.definedata("y");  * Pull in variable Y from this table;
        h.definedone();
    end;

    * Set WORK.CLEAN_1 data;
    * This acts as the base data;
    set work.clean_1;

    * Initialize values of Y as missing;
    call missing(y);

    * Match rows based on the defined key (ID) using the H hash object;
    rc = h.find();

    * Write matched rows only to output data table;
    * For hash objects, a return code of 0 indicates a match;
    if rc = 0 then output;

    * Drop the temporary return code variable from output table;
    drop rc;
run;

title "Hash Object Merge: Inner Join";
proc print data=work.merge_hash_clean_1;
run;
title;

* Left join: keep all rows in base (or left) table ----------------------------;
data work.merge_hash_clean_2;
    if 0 then set work.clean_1 work.clean_2;

    if _n_ = 1 then do;
        declare hash h(dataset: "work.clean_2", duplicate: "error");
        h.definekey("id");  * Merge key variable;
        h.definedata("y");  * Pull in variable Y from this table;
        h.definedone();
    end;

    set work.clean_1;

    call missing(y);

    rc = h.find();

    * By omitting the following, all rows of WORK.CLEAN_1 are output;
    /* if rc = 0 then output; */

    * Display the return code this time;
    /* drop rc; */
run;

title "Hash Object Merge: Left Join";
footnote "Return code (RC) = 0 indicates successful match";
proc print data=work.merge_hash_clean_2;
run;
title;
footnote;

The examples above demonstrate inner- and left joins using hash objects. These are the most common join types I use in my day-to-day work, and hash object merging works well for both of these.

Hash object merging generally is predicated on comparing a base table to one or more hash objects. The user may seek to merge values from the hash object onto the base table, or perhaps filter rows in the base table based on the hash object. An outer join, on the other hand, would involve adding rows that appear only in the hash object to the final, merged output table. This is a little outside of the hash object paradigm, so I have not provided such an example here. Generally, in the exceedingly rare cases in which I need to do an outer join, I use one of the traditional methods.

Many-to-Many Merge

The following argument in the hash object declaration is key.

duplicate: "error"

If the variable (or combination of variables) in the definekey statement is/are not unique, the above argument tells SAS to stop processing and throw an error. This failsafe provides the user with helpful feedback if he/she assumes the merge key is unique and that assumption is violated. Let's look at an example of what happens when the merge key is not unique.

data work.merge_hash_dup_1;
    if 0 then set work.dup_1 work.dup_2;

    if _n_ = 1 then do;
        declare hash h(dataset: "work.dup_2", duplicate: "error");  * Throw error if duplicate key;
        h.definekey("id");
        h.definedata("y");
        h.definedone();
    end;

    set work.dup_1;

    call missing(y);

    rc = h.find();

    * Write matched rows only to output data table;
    if rc = 0 then output;

    drop rc;
run;
sas

Hash Objects: Pros

I find hash objects to be incredibly useful because I regularly merge large sets of data that reside in multiple tables. Here are a few of my favorite aspects of hash objects.

Goodbye Sorting

Within the context of SAS 9, sorting tables for the mere purpose of merging them adds code that arguably serves no standalone purpose beyond satisfying a prerequisite to permit BY group processing for the MERGE statement. PROC SQL reduces code glut by eliminating the need for PROC SORT prior to the join, but PROC SQL does not eliminate the actual sorting, it just quietly handles that for the user. Sorting can be a very computationally expensive process that slows down workflows that include large data tables. CAS data on SAS Viya does not have to be sorted prior to a DATA step merge, but the chaotic many-to-many merge outcomes remain a risk, and PROC SQL is not permitted to read from CAS and write back to CAS in the same step at present.

In contrast to the traditional approaches, hash object merging requires no prior sorting, and it does not sneakily perform a sort under the hood without informing the user, either. Further, hash object merges work well in both SAS 9 and SAS Viya.

Improved Transparency

In my experience providing debug assistance to novice SAS coders, the overlooked many-to-many merge is second only to the misplaced semicolon among those head-scratching "I don't know what went wrong" problems they sometimes face. Less experienced SAS coders may simply glance at the log and ensure all the text is blue, if they look at all. The DATA step is quite meek in reporting a many-to-many merge in the log, and PROC SQL says nothing at all about it. Many-to-many merges may wreak havoc on workflows, and this issue may be difficult to find in the log, too. Allowing the user to see an error message when a many-to-many merge occurs is an underrated advantage of hash object merging.

Performance and Extended Use Cases

Perhaps best of all, hash object merges often execute much faster than traditional approaches. I did not provide advanced examples here, but hash objects can be used in powerful ways to elegantly tackle difficult use cases. For example, it is straightforward to slightly extend the code in this notebook to merge and/or filter three or more tables all in one step with hash objects. This notebook just scratches the surface, but hash objects are handy for a variety of purposes.

Hash Objects: Cons

Hash objects are very powerful and useful, but they do have downsides. I will discuss a couple of key limitations that are worth considering.

The Code

This may just be my personal taste, but I find the syntax for SAS hash objects to be a bit clunky and even counterintuitive in certain regards. The user is asked to manually perform certain tasks that SAS usually does automatically. Loading variable characteristics into the program data vector (PDV) in that odd-looking if 0 then set...; statement is not typically required in a run-of-the-mill DATA step. Similarly, initializing variables with missing values ahead of a merge is also a task that a user need not worry about under more traditional approaches.

In some regards, the syntax for hash objects may seem to break with established SAS coding conventions. DATA or PROC step references to data sets and variable names do not often involve quotes or commas, but they are mandatory when declaring a hash object. Even the hash object return code may be counterintuitive. Although SAS does not have a dedicated Boolean data type, a value of 0 corresponds to "false" in standard SAS logic, but in the case of a hash object lookup, a value of 0 indicates a successful key match (i.e., "true").

Table Size Limitations

In traditional SAS 9 data processing, a data table is read from disk and written back to disk one row at a time. This permits SAS 9 to make use of fairly large data tables, even on modest hardware, because there is no need to fit the entire data table into memory... except when dealing with hash objects. The entirety of the hash object's data is loaded into memory. Although this significantly speeds up performance and gets around the need to sort two tables that are saved to disk, the execution environment must have sufficient available memory to hold all of the hash object's data, or else the step will fail. Fortunately, more modern SAS architecture—SAS Viya in particular—tends to be sized such that available memory is rarely, if ever a concern. This limitation mainly comes into play when executing on less capable hardware or utilizing exceptionally large data tables, in which case traditional read/write-based sorting may still be necessary.

Final Thoughts

Hash objects provide a powerful means to merge tables efficiently and with improved confidence in the integrity of the merge. Although modern implementations such as SAS Viya may address certain pain points with traditional methods, such as eliminating the sorting requirement and improved execution speed, hash object merging remains a fast, flexible, and relevant approach.
